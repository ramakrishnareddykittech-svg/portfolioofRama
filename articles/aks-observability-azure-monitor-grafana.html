AKS observability

Operating AKS with Azure Monitor and Grafana after the deployment succeeds.

How to turn cluster, workload, dependency, and release data into dashboards and alerts that help during a real incident.

By Ramakrishna Reddy Kandala 9 minute read AKS · Azure Monitor · Prometheus · Grafana

A green pipeline is not production health

A pipeline can report success while users receive errors, pods restart, dependencies time out, or latency grows. Deployment status is one signal. Operations needs a connected view of the platform, the application, its dependencies, and the release that changed them.

Organize signals into four layers

1. Cluster and node health

Track node readiness, CPU and memory pressure, disk pressure, scheduling failures, network problems, and autoscaler behavior. These signals answer whether the platform can run the requested workloads.

2. Kubernetes workload health

Track desired versus available replicas, pending pods, restart counts, OOM kills, failed probes, deployment rollout duration, and namespace resource usage.

3. Application and dependency health

Track request rate, error rate, latency, saturation, queue depth, database calls, external API failures, and business-level success signals. Container health alone cannot tell whether a payment, order, or login flow works.

4. Release health

Add deployment annotations or variables to dashboards so an error spike can be compared with the image tag, pipeline run, commit, and Helm revision that introduced it.

Use each tool for the job it does well

  • Azure Monitor and Log Analytics: Azure resource context, activity, inventory, container logs, and Kusto-based investigation.
  • Prometheus: time-series workload and Kubernetes metrics with flexible labels.
  • Grafana: shared operational dashboards across platform and application signals.
  • Application Insights: request traces, exceptions, dependencies, and application performance.

The goal is not to duplicate every signal in every tool. Define which source is authoritative for each operational question.

Build dashboards around decisions

A dashboard should help someone decide what to do next. A useful AKS service view can answer:

  • Are users affected?
  • Did the problem start after a deployment?
  • Is the failure isolated to one pod, node, namespace, or region?
  • Is the application failing, or is a dependency slow?
  • Can the workload scale, or is it blocked by resource limits?
  • Should the release continue, pause, or roll back?
Avoid dashboards made only of infrastructure utilization. CPU can look normal while users receive errors. Always include service-level behavior.

Create alerts that lead to action

Alerts should be specific enough to act on and stable enough to trust. Every production alert needs an owner, a severity, a clear condition, useful context, and a runbook.

Examples include sustained error-rate increase, unavailable replicas, repeated OOM kills, a deployment exceeding its expected rollout time, or a critical dependency crossing its latency objective.

Do not alert separately on every noisy symptom. Combine related signals and use a time window that avoids paging on short, harmless transitions.

Keep logs queryable

Structured logs should include a timestamp, severity, service, environment, request or correlation ID, and a safe error description. Avoid logging secrets, tokens, connection strings, or personal data. 

{
  "level": "error",
  "service": "orders-api",
  "environment": "prod",
  "correlationId": "7f8b...",
  "event": "dependency_timeout",
  "dependency": "inventory-api"
}

Correlation IDs connect ingress, application, and dependency events. Without them, incident investigation becomes a manual search across unrelated timestamps.

Observe the rollout itself

During a release, watch available replicas, pod readiness, restart count, error rate, latency, and dependency behavior. Compare the new and previous revision where possible.

A release gate should fail when health moves outside agreed limits. Waiting for users to report a problem is not a validation strategy.

Write the runbook while the context is fresh

A useful runbook includes the dashboard link, common failure patterns, safe diagnostic commands, rollback steps, dependency contacts, and conditions that require escalation. Keep it close to the service and review it after incidents.

Production handoff questions

  • Who owns each alert during and outside business hours?
  • Which dashboard is the starting point for this service?
  • How long are logs and metrics retained?
  • Can an operator identify the deployed image and Helm revision?
  • What is the rollback command and when should it be used?
  • Which signals prove that users recovered after mitigation?

The operating principle

Observability is complete when the team can detect a meaningful problem, understand its scope, connect it to a change, choose a safe response, and confirm recovery. Tools matter, but ownership and decision-ready signals are what make the platform operable.
