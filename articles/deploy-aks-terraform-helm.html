AKS + Infrastructure as Code

Deploying AKS with Terraform and Helm without creating release chaos.

A practical delivery model for separating platform provisioning from workload releases while keeping identity, rollback, and observability connected.

By Ramakrishna Reddy Kandala 10 minute read Azure · AKS · Terraform · Helm

Start with a clean ownership boundary

Terraform and Helm solve different problems. Terraform should own Azure resources and their access relationships. Helm should own Kubernetes workload configuration. Mixing both layers into one large deployment script makes failures harder to understand and rollback.

A practical split is:

  • Terraform: resource groups, VNet, subnets, AKS, ACR, Key Vault, Log Analytics, identities, and role assignments.
  • Helm: Deployments, Services, Ingress, ConfigMaps, probes, autoscaling, resource settings, and application release values.

Design Terraform outputs for the release layer

The application pipeline should not rediscover infrastructure by guessing resource names. Terraform outputs can expose only the values needed by later automation, such as the cluster name, resource group, registry login server, and Key Vault name. 

output "aks_cluster_name" {
  value = azurerm_kubernetes_cluster.platform.name
}

output "acr_login_server" {
  value = azurerm_container_registry.platform.login_server
}

output "key_vault_name" {
  value = azurerm_key_vault.platform.name
}

Outputs are a contract. Avoid exposing secrets, kubeconfig content, or credentials through them.

Prefer identity over stored credentials

The release process needs permission to publish images and deploy workloads, but the running application should not inherit those permissions. Keep deployment identity and workload identity separate.

  • Grant the AKS kubelet identity permission to pull from ACR.
  • Use workload or managed identity for application access to Azure services.
  • Use Key Vault for secrets that cannot be replaced with identity.
  • Scope pipeline permissions to the target environment and required actions.
A pipeline that can deploy to production should not automatically have broad subscription ownership. Convenience is not a valid reason for permanent high privilege.

Publish immutable images

Do not promote a moving tag such as latest. Use a commit SHA, build ID, or release identifier so every Kubernetes revision maps back to one image and one source change. 

image:
  repository: myregistry.azurecr.io/orders-api
  tag: "a81f0c9"
  pullPolicy: IfNotPresent

Promotion should change the approved image tag, not rebuild the same source differently for each environment.

Make health checks part of release design

Kubernetes cannot protect a rollout if the application exposes no useful health endpoints. Readiness should answer whether a pod can receive traffic. Liveness should answer whether the process is stuck and needs restarting. 

readinessProbe:
  httpGet:
    path: /health/ready
    port: http
  initialDelaySeconds: 10
  periodSeconds: 5

livenessProbe:
  httpGet:
    path: /health/live
    port: http
  initialDelaySeconds: 30
  periodSeconds: 10

Avoid making liveness depend on every external service. A temporary database issue should not necessarily restart every application pod.

Use environment values without copying charts

Keep one reusable chart and separate value files for development, test, and production. Differences should be intentional and reviewable: replica counts, resource settings, hostnames, autoscaling thresholds, and environment references.

If each environment has its own copied chart, fixes eventually land in one place and not another.

Gate the rollout with real checks

  1. Run unit tests and static checks.
  2. Build the image once.
  3. Run the selected quality or security checks.
  4. Push the immutable image to ACR.
  5. Render or lint the Helm chart.
  6. Deploy with a timeout and atomic rollback behavior.
  7. Check rollout status and application health.
  8. Record the deployed image and Helm revision.
helm upgrade --install orders-api ./charts/orders-api \
  --namespace orders \
  --create-namespace \
  --values ./environments/prod.yaml \
  --set image.tag="$BUILD_COMMIT" \
  --atomic \
  --timeout 10m

Production handoff checklist

  • Dashboards show pod health, errors, latency, saturation, and dependency failures.
  • Alerts identify an owner and link to a useful runbook.
  • Resource requests and limits come from observed workload behavior.
  • Rollback has been exercised with a known previous image.
  • Terraform state and deployment logs are protected and auditable.
  • Someone outside the implementation team can operate the release.

The practical result

A strong AKS delivery model is not just a cluster and a pipeline. It is a chain of explicit contracts: Terraform owns the platform, Helm owns workload configuration, identity controls access, immutable images preserve traceability, and observability confirms whether the release is actually healthy.
